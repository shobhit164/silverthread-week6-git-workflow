Git Collaboration Workflow

Shobhit, Nikita, and Payal used a shared GitHub repository to demonstrate how branches, reviews, merges, and conflict resolution help developers work in parallel.

Repository Link: https://github.com/shobhit164/silverthread-week6-git-workflow.git
By - Silverthread
Shobhit Owned the product card layout branch and reviewed Payal's checkout bugfix.
Nikita Owned the promo banner branch, added the sale message, and reviewed Shobhit's product card layout for clarity.
Payal Owned the checkout bugfix branch and reviewed Nikita's promo banner update.

Branching Model

main Stable shared branch with reviewed code.
Shobhit: feature/product-card-layout Improved product card layout and inventory display.
Nikita: feature/promo-banner Added seasonal sale messaging.
Payal: bugfix/checkout-button-copy Fixed unclear checkout button text.

Team Workflow

  1. Create a branch from `main` for one focused task.
  2. Commit the task with a clear message.
  3. Open a pull request and ask Shobhit, Nikita, or Payal to review.
  4. Merge approved work into `main`.
  5. Resolve conflicts when branches edit the same file.

Workflow Timeline

01Create shared GitHub repository and invite teammates.
02Create separate task branches from main.
03Commit focused UI and logic changes independently.
04Open pull requests and review each other's work.
05Merge approved PRs and resolve the final conflict.

Conflict Resolved

Two branches edited `src/index.html`: Nikita's branch added a weekend sale banner, while Payal's branch added a free-shipping message and checkout copy fix.

Final decision: combine both ideas into one banner and keep the improved product layout plus the clearer button text.

Integrated Changes

  • Product card now shows a cleaner price and inventory row.
  • Promo banner now communicates both discount and shipping offer.
  • Checkout button now uses clearer action-focused text.
  • Documentation records reviews, conflict, and resolution.

Lessons Learned

  • Small branches made each teammate's work easier to review.
  • Pull requests helped the team discuss changes before merging.
  • Merge conflicts were easier to solve when the team compared both versions carefully.
  • GitHub history made it clear who changed what and when.
